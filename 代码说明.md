## 1. 项目定位与用户故事

### 1.1 产品定位

**舆情爬虫 v9** 是一个 Flask + Patchright（CDP 控制 Chrome）的本地 Web 应用，用于：

1. **手工爬取**：黑猫投诉、小红书关键词搜索 + 详情抓取  
2. **合作方风险情报**：按合作方名单自动监测多源舆情 → AI 打标 → 看板/API 交付  

业务系统对 `source` 自行加权；本系统提供高召回 `relevance`、情感分、风险类型与导出。

### 1.2 用户故事（端到端）

| ID | 角色 | 故事 | 涉及模块 |
|----|------|------|----------|
| US-01 | 运营 | 启动 Chrome，在黑猫/小红书搜索关键词并导出 CSV/JSON | `/?tab=crawl`, `crawler_web.py`, `reports.py` |
| US-02 | 运营 | 未登录时勾选「爬取详情」，系统暂停并等待扫码，登录后自动续跑 | `login_gate.py`, `auth_utils.py`, `heimao_session.py` |
| US-03 | 运营 | 小红书详情通过**搜索页弹窗**抓取，避免 App 墙 | `xhs_detail.py`, `crawl_xhs` |
| US-04 | 风控 | 维护合作方名单（名称、别名、排除词、监测关键词） | `/?tab=partners`, `intel/db.py`, `/api/partners`（写需管理员） |

#### 调度

- `intel/scheduler.py`：APScheduler 读 `monitor_tasks.schedule_json.cron` 定时触发同一 task（增量 run）  
- 配置：`monitor.scheduler_enabled`、`monitor.scheduler_timezone`

### 3.8 `intel/runner.py`


## 4. 数据流（监测任务）

### 4.1 源级 crawl_mode（任务级字段已降级）

| 源 | `config.sources.{id}.crawl_mode` | routine 行为 |
|----|----------------------------------|--------------|
| **xhs** | 强制 `list_first` | 关键词 batch 列表爬取，routine **零弹窗** |
| **heimao** | 默认 `legacy` | 合作方 × 关键词 legacy 爬取，可选 `fetch_detail` |

- `monitor_tasks.crawl_mode` **保留列**（schema 迁移默认 `legacy`），语义：
  - **混合源任务**（含 xhs 或多源）：**忽略** task 级值，改读各源 `config.sources.*.crawl_mode`
  - **仅 heimao 单源** 且 `monitor.workers.enabled=false`：仍可用 task.crawl_mode 作 fallback
- UI 任务编辑页说明「爬取策略由数据源配置决定」

### 4.2 单进程路径（`monitor.workers.enabled=false`，默认）

```
合作方名单(partners)
    ↓
创建 monitor_task(partner_ids, sources, max_pages)
    ↓ Run 开始前 per-source diagnose（partial 降级）
混合源串行：heimao legacy 片段 + xhs list_crawl 片段
    ↓ list_triage（仅 crawl_phase=list 的 raw）
    ↓ build_investigation_queue → investigation crawl
    ↓ AnalyzePipeline（ThreadPool parallel_batches）
看板 / GET /api/intel/records / export
```

### 4.3 多 Worker 路径（`monitor.workers.enabled=true`）

```
Orchestrator (runner.py)
  ├─ enqueue crawl_work_queue（heimao: legacy_crawl ∥ xhs: list_crawl）
  ├─ spawn Worker 子进程（各实例独立 CDP 端口 + profile + cookies_file）
  ├─ barrier routine → list_triage → enqueue investigation（按 source 回派 Worker）
  └─ analyze（Orchestrator，parallel_batches 默认 5）
```

- Run 并发控制：`monitor_task_runs.status` + `stop_requested`（**非**仅 `S.running`）
- xhs investigation：`max_modal_per_run` Run 级配额，超限 skip 不 fail run
- 回滚：设 `monitor.workers.enabled=false` 恢复单进程

#### 监测任务 Tab — Run 历史 UI（`static/panel-intel.js`）

- 任务行点击「历史」：行内展开 Run 摘要表，默认 `GET .../runs?page=1&limit=5`
- 「加载更多」递增 page 追加较旧记录；显示 `已加载 x / total`
- 点击摘要行：`UiShell.drawer` 展示完整 stats（含 label + help）、分源 timing/token、错误信息与 `<details>` 字段 glossary
- 支持 `?tab=tasks&run_id=` 深链；首页最近 Run 点击同样打开 Drawer
- 合作方/任务创建编辑使用 Modal；主要交互使用 Toast / Confirm 替代 `alert()`/`confirm()`

## 8. 已知限制与待办

| 项 | 说明 |
|----|------|
| 手工 crawl 与 Worker 争用 CDP | Run 进行中或端口被 Worker 占用时，`/api/crawl_*` 返回 **409** |

### 8.1 非回归约束（parallel-crawl-workers-selective-xhs）

以下行为在 Worker / 并行 analyze 引入后 **必须保持**：

| 能力 | 说明 |
|------|------|
| `early_stop` | heimao / xhs 分页见底早停逻辑与现网一致 |
| `task_timeout_sec` / `analysis_timeout_sec` / `min_crawl_timeout_sec` | 分段超时：`crawl_budget = task - analysis_reserve`，且 crawl ≥ min_crawl；`analysis_timeout` 不得接近 `task_timeout`（否则爬取被压至 5 分钟）。失败 `progress.reason`：`crawl_timeout` / `timeout` |
| `skip_if_running` / scheduler | 改查 `monitor_task_runs` active 状态，语义仍为 skipped_overlap |
| incremental analyze | dedup、payload 变化检测、full_replace 行为不变 |
| xhs investigation | 禁止 goto explore；弹窗路径不变 |
| `match_all_partners` | 多方匹配逻辑不变 |
