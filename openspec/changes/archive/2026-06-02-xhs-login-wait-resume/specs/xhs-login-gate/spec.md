## 新增需求

### 需求：爬取详情须已登录

当用户以启用 `fetch_detail` 的方式启动小红书爬取时，系统必须在打开任何笔记详情页之前将已认证会话视为前置条件。

#### 场景：任务开始时未登录

- **当** 以 `fetch_detail=true` 启动小红书爬取
- **且** 在加载搜索页之前登录检测失败
- **则** 爬取进入 `WAITING_LOGIN` 阶段
- **且** 在登录成功前系统不得打开任何详情页

#### 场景：仅抓列表、不抓详情

- **当** 以 `fetch_detail=false` 启动小红书爬取
- **且** 登录检测失败
- **则** 系统必须继续列表抓取，且不得进入 `WAITING_LOGIN`

### 需求：等待用户登录后续跑

当小红书处于 `WAITING_LOGIN` 时，系统必须保持 CDP 浏览器会话，并轮询登录状态直至成功或超时。

#### 场景：等待期间用户完成登录

- **当** 爬取处于站点 `xhs` 的 `WAITING_LOGIN`
- **且** 用户在已连接的 Chrome 窗口中完成登录
- **且** 下一次登录轮询成功
- **则** 爬取必须从暂停处续跑，且无需新的爬取请求

#### 场景：登录等待超时

- **当** 爬取处于站点 `xhs` 的 `WAITING_LOGIN`
- **且** 在配置超时内登录未成功
- **则** 爬取必须在日志与状态中明确失败停止
- **且** 等待前已收集的部分列表结果可以保留

### 需求：小红书登录检测

系统必须使用可配置规则（含会话 Cookie 与登录墙页面文案）判定小红书登录状态。

#### 场景：有效会话 Cookie

- **当** 浏览器上下文中存在已配置 Cookie 名的非空值（默认 `web_session` 与 `webId`）
- **则** 登录检测必须通过

#### 场景：页面出现登录墙

- **当** 页面正文包含任一配置的登录失败文案（如「登录后查看」）
- **则** 登录检测必须失败

#### 场景：打开笔记时的详情探测

- **当** `fetch_detail=true`
- **且** 已打开详情页
- **且** 详情提取的主正文为空
- **且** 页面表明需要登录
- **则** 系统必须将当前会话视为未认证并进入 `WAITING_LOGIN`

### 需求：API 与 UI 可观测的等待状态

系统必须向 Web UI 与状态 API 暴露等待登录状态。

#### 场景：等待期间的状态

- **当** 小红书爬取处于 `WAITING_LOGIN`
- **则** `/api/status` 必须标明 `phase=waiting_login`（或等价字段）
- **且** 包含站点、已用时间与超时配置

#### 场景：用户引导

- **当** 为小红书进入 `WAITING_LOGIN`
- **则** 系统必须在日志中提示在 Chrome 内完成登录
- **且** UI 必须显示任务正在等待登录（而非静默空闲）

### 需求：等待成功后可选导出 Cookie

在等待后检测到登录成功时，系统可以将 Cookie 持久化到配置的凭证文件，以改善后续运行。

#### 场景：续跑后自动导出

- **当** 登录等待成功结束
- **且** `auth.xhs.auto_export_after_login` 为 true（若已配置则默认为 true）
- **则** 系统必须将小红书 Cookie 导出到 `credentials/xhs_cookies.json` 或配置路径
